# MAC 位址相同時，完整指紋碰撞的可能性分析

---

## 核心問題

「如果兩台機器的 MAC 位址相同，它們的完整指紋會一樣嗎？」

簡答：**不會，除非其他所有識別碼也同時相同。**

指紋是所有識別碼組合後做 SHA-256，只要有任何一個不同，指紋就不同。
問題的關鍵因此轉變成：**MAC 相同的情況下，其他識別碼也會同時相同的機率有多高？**

以下逐一拆解 MAC 位址相同的每種來源，並分析各情境下的指紋碰撞風險。

---

## 一、MAC 位址結構回顧

```
MAC 位址：48 bits
┌──────────────────────┬──────────────────────┐
│  OUI（廠商識別碼）    │  裝置序號（NIC ID）   │
│     24 bits          │      24 bits          │
│  由 IEEE 分配給廠商   │  由廠商自行管理       │
└──────────────────────┴──────────────────────┘

例：  00:1A:2B  :  3C:4D:5E
      └── Intel  └── 第 3,997,150 片網卡
```

理論上 24 bits 的裝置序號能區分 16,777,216 片網卡。
但在現實中有多種情境會讓 MAC 重複。

---

## 二、MAC 相同的六種情境與指紋碰撞分析

---

### 情境一：出廠序號管理失誤（OEM 重複燒錄）

**發生原因：**

網卡製造商或主機板 OEM 在大量生產時，因自動化燒錄程式的 bug、
批次編號計數器溢位、或品管疏失，將相同的 MAC 位址燒入多片網卡。

這並非罕見事件，歷史上有明確紀錄的案例：

| 時間 | 廠商 | 問題 |
|------|------|------|
| 2012 | 某台灣主機板品牌 | 同批次主機板 Wi-Fi 模組 MAC 重複 |
| 2018 | 某工控網卡廠 | OUI 計數器未重置，同型號產品 MAC 全為 `00:e0:4c:68:02:00` |
| 多次 | 廉價 USB 網卡 | 批量使用同一測試 MAC `00:e0:4c:00:00:00` 出貨 |

**此情境下指紋碰撞風險：**

```
MAC 相同（出廠重複燒錄）
  ↓
Windows MachineGuid：各自獨立安裝 → 不同  ✓
Linux machine-id：各自獨立安裝 → 不同    ✓
macOS IOPlatformUUID：出廠燒錄不同 → 不同 ✓
BIOS UUID：各主機板序號不同 → 不同        ✓

碰撞結論：指紋不同，授權系統不受影響
```

MAC 重複，但其他三層都不同，SHA-256 結果完全不同。
**此情境對本授權系統無影響。**

---

### 情境二：使用者或管理員手動設定相同的 MAC

**發生原因：**

- 管理員為了方便管理或測試，把多台機器的 MAC 設成同一個固定值
- 企業政策要求特定 VLAN 的機器使用固定 MAC 格式
- 攻擊者刻意複製授權機器的 MAC（主動仿冒）

**此情境下指紋碰撞風險：**

```
手動設定相同 MAC（非主動仿冒）：
  MAC 相同
  MachineGuid / machine-id：各自安裝，不同 ✓
  BIOS UUID：不同主機板，不同 ✓

  碰撞結論：指紋不同 ✓


手動設定相同 MAC（主動仿冒，攻擊者已知道目標 MAC）：
  MAC 相同（攻擊者複製）
  MachineGuid：若攻擊者也複製 → 相同 ✗
  BIOS UUID：攻擊者未必知道 → 不同 ✓（有 BIOS 層的情況下）

  碰撞結論：加入 BIOS UUID 後，仿冒者需同時知道並複製三個值
```

這正是加入 BIOS UUID 層的直接效益：讓攻擊成本從「複製 2 個值」變成「複製 3 個值」。

---

### 情境三：虛擬機使用相同的 MAC 設定

**發生原因：**

VMware / VirtualBox / Hyper-V 在建立 VM 時，若：
- 使用固定 MAC 設定（而非隨機產生）
- 多個 VM 都指定了同一個固定 MAC
- VM 範本複製後 MAC 設定被一起複製

**此情境下指紋碰撞風險（以 VMware 為例）：**

```
VM-A（clone 自同一範本，MAC 相同）
VM-B

  MAC：相同（範本 clone，未重設 MAC）
  MachineGuid：若也沒執行 sysprep → 相同 ✗
  BIOS UUID：VMware 的虛擬 BIOS UUID 在 clone 時也被複製 → 相同 ✗
  machine-id（Linux）：若也沒重設 → 相同 ✗

  碰撞結論：⚠️ 全部相同，指紋完全碰撞
```

**這是真正需要關注的情境。**

VM clone 不執行任何重設步驟時，三層識別碼（MAC + 系統ID + BIOS UUID）
都會相同，指紋完全碰撞，等於一份授權可以用在所有 clone 的 VM 上。

**緩解方式：**

```bash
# VMware：clone 後在 .vmx 檔中刪除 MAC 設定，讓 VMware 重新產生
# 或在 VM 設定介面選「Generate new MAC address」

# Windows（clone 後執行）
sysprep /generalize /oobe /shutdown

# Linux（clone 後執行）
rm /etc/machine-id && systemd-machine-id-setup
```

---

### 情境四：Docker / 容器的 MAC 碰撞

**發生原因：**

Docker 預設使用 `172.17.0.0/16` 子網，容器的 veth 介面 MAC 由核心產生，
但若多個容器都使用 `--mac-address` 指定同一個值，MAC 相同。

**此情境下指紋碰撞風險：**

```
容器A（--mac-address 02:42:ac:11:00:02）
容器B（--mac-address 02:42:ac:11:00:02）

  MAC：相同
  machine-id：繼承自宿主機 → 相同 ✗
  BIOS UUID：容器內無法讀取 dmidecode → 採集失敗，不加入

  碰撞結論：⚠️ MAC + machine-id 相同，指紋碰撞
```

這也是文件中建議「授權應綁定宿主機而非容器」的原因之一。

---

### 情境五：Wi-Fi MAC 隨機化後的意外碰撞

**發生原因：**

現代作業系統（Windows 10+、Android 10+、iOS 14+）預設開啟 Wi-Fi MAC 隨機化，
每次連線都產生不同的隨機 MAC。

這對本系統的影響不是碰撞，而是**不穩定**：
每次重新連線 Wi-Fi，`uuid.getnode()` 回傳的 MAC 可能不同，導致指紋改變。

**此情境下的指紋穩定性：**

```
第一次連線：MAC = aa:bb:cc:dd:ee:ff → 指紋 A
斷線後重新連線：MAC = 11:22:33:44:55:66 → 指紋 B ← 授權失效！
```

**緩解方式：**

`uuid.getnode()` 在 Python 內部會嘗試找最穩定的網路介面，
通常優先回傳有線網路（乙太網路）的 MAC，
只有在**完全沒有有線網路介面**時才會使用無線介面。

若甲方機器是純 Wi-Fi 環境（無有線網路），建議在採集時明確指定有線介面，
或完全不依賴 MAC 位址（讓 MachineGuid + BIOS UUID 提供足夠的唯一性）。

---

### 情境六：同一個網段的 MAC 廣播位址 / 多播位址碰撞

**技術說明：**

MAC 位址第一個 byte 的最低位（bit 0）為 1 時，代表多播位址（Multicast），
第二低位（bit 1）為 1 時代表本地管理位址（Locally Administered）。

`uuid.getnode()` 已過濾這些情況（若回傳 `0` 或 `2^48-1` 則跳過），
不會誤採集廣播或多播位址。

此情境對本系統無影響。

---

## 三、各情境碰撞風險總覽

| MAC 相同的原因 | 其他識別碼是否也相同 | 指紋碰撞風險 | 影響授權系統 |
|--------------|-------------------|------------|------------|
| OEM 出廠燒錄重複 | 否（各自安裝） | 無 | 無影響 |
| 管理員手動設相同值 | 否（各自安裝） | 無 | 無影響 |
| 主動仿冒（複製 MAC） | 需同時複製其他值 | 低（需 3 個值） | 有限 |
| VM clone 未重設 | **是（全部複製）** | **極高** | **完全碰撞** |
| 容器指定相同 MAC | machine-id 繼承宿主機 | 高 | 碰撞 |
| Wi-Fi MAC 隨機化 | — | — | 不穩定（非碰撞） |

---

## 四、加入 BIOS UUID 後的碰撞防護效果

```
攻擊者需要同時偽造的欄位（加入 BIOS UUID 前後比較）：

加入前：
  mac:<值>  +  wguid/mid/ioplatform:<值>
  → 需偽造 2 個值

加入後：
  mac:<值>  +  wguid/mid/ioplatform:<值>  +  biosuuid:<值>
  → 需偽造 3 個值，且 BIOS UUID 來自不同層（韌體），
    不會因為 OS 映像相同而自動相同
```

對於 VM clone 的情境，BIOS UUID 的效益取決於 Hypervisor：

| Hypervisor | Clone 後 BIOS UUID 是否相同 |
|------------|---------------------------|
| VMware Workstation | 相同（需手動重設） |
| VMware ESXi | 視 clone 設定，可設定為自動重新產生 |
| VirtualBox | 相同（需手動執行 `vboxmanage modifyvm --hardwareuuid`） |
| Hyper-V | **不同**（每個 VM 自動產生新 GUID） |
| KVM/QEMU | 視 domain XML 設定 |

---

## 五、結論

**MAC 位址相同，指紋不一定相同。**
只有當所有採集到的識別碼（系統ID + BIOS UUID + MAC）全部相同時，
指紋才會碰撞。在正常的獨立安裝環境下，即使 MAC 重複，
其他兩層提供了足夠的獨立性。

**真正需要擔心的唯一情境是 VM/容器未執行重設步驟的 clone。**
這個問題不是演算法的缺陷，而是部署流程的問題，
解決方式是要求甲方在 clone 後執行對應的重設指令。

**本系統當前（加入 BIOS UUID 後）的防護邏輯：**

```
指紋 = SHA-256(
  sorted([
    "mac:xx",           ← 容易改，輔助用途
    "wguid/mid/io:xx",  ← 需 admin/root 才能改，主要識別碼
    "biosuuid:xx",      ← 來自韌體，映像複製不影響（Hyper-V/實體機）
  ])
)

攻擊者若想仿冒，需同時知道並複製三個來自不同層的值。
SHA-256 的單向性確保他無法從 license.lic 的指紋字串反推任何一個原始值。
```
